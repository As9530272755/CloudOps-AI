# 多用户管理实战示例

## 场景

公司：3 个团队，50+ 用户，20+ 集群

| 团队 | 人数 | 职责 |
|------|------|------|
| 平台部 | 5 人 | 平台管理、集群接入、全局配置 |
| 运维部 | 10 人 | 日常运维、故障处理、发布变更 |
| 业务组 | 35 人 | 查看自己业务 Pod、日志、排查问题 |

集群：
- 生产集群 A/B/C（8 个）：谨慎，目前只配 viewer 凭证
- 预发集群 D/E（4 个）：配 viewer + operator
- 测试集群 F/G/H（8 个）：配 viewer + operator + admin

---

## 方案一：逐个用户授权（适合 50 人以下）

### Step 1：创建 3 个角色模板

```
平台管理员 (platform-admin)
  └── 权限：全部
  └── 凭证级别：admin

运维工程师 (cluster-operator)
  └── 权限：集群查看 + 工作负载操作 + 日志 + Terminal
  └── 凭证级别：operator

业务开发 (cluster-viewer)
  └── 权限：只读（查看 Pod/Deployment/日志/事件）
  └── 凭证级别：viewer
```

### Step 2：给用户授权集群

**张三（平台部负责人）**
```
用户：张三
角色：平台管理员
授权集群：
  - 生产集群A → admin（operator 凭证兜底）
  - 生产集群B → admin
  - 生产集群C → admin
  - 预发集群D → admin
  - 预发集群E → admin
  - 测试集群F → admin
  ...（全部 20 个集群）
```

**李四（运维组长）**
```
用户：李四
角色：运维工程师
授权集群：
  - 生产集群A → operator（viewer 凭证，operator 未配置则降级）
  - 生产集群B → operator
  - 生产集群C → operator
  - 预发集群D → operator
  - 预发集群E → operator
  - 测试集群F → operator
```

> 生产集群目前只配了 viewer 凭证，所以李四在生产集群A 会自动降级为 viewer，前端提示：
> "该集群未配置 operator 凭证，已降级为只读。请联系管理员添加。"

**王五（业务开发）**
```
用户：王五
角色：业务开发
授权集群：
  - 生产集群A → viewer
  - 预发集群D → viewer
  - 测试集群F → viewer
```

### Step 3：后续升级生产集群

**2 周后，生产集群A 需要支持运维操作：**

管理员操作：
1. 进入"集群管理 → 生产集群A → 凭证管理"
2. 点击"添加凭证"
3. 级别选 `operator`，粘贴 operator 级别的 kubeconfig
4. 保存

**效果（无需改代码，无需重新授权用户）：**
- 张三 → 生产集群A → 自动使用 admin 凭证（如果有）或 operator 凭证
- 李四 → 生产集群A → 之前降级为 viewer，现在自动升级到 operator
- 王五 → 生产集群A → 仍然是 viewer（角色限制，不升级）

## 方案二：用户组批量授权（适合 50+ 人，推荐）

### 问题

50 个用户 × 20 个集群 = 1000 条授权记录，逐个配置累死。

### 解决：引入"用户组"概念

```
用户组：平台部-admin
  └── 成员：张三、赵六、孙七（5 人）
  └── 角色：平台管理员
  └── 授权集群：全部 20 个集群

用户组：运维部-operator
  └── 成员：李四、周八、吴九（10 人）
  └── 角色：运维工程师
  └── 授权集群：全部 20 个集群

用户组：业务组-viewer
  └── 成员：王五、郑十 等（35 人）
  └── 角色：业务开发
  └── 授权集群：仅测试集群 F/G/H（3 个）
```

**效果：**
- 新增用户 → 加入对应用户组 → 自动继承组的授权
- 调整权限 → 改组的授权 → 全组成员生效
- 用户调岗 → 从 A 组移除，加入 B 组 → 权限自动切换

### 数据库设计（扩展）

```go
type UserGroup struct {
    ID            uint
    TenantID      uint
    Name          string
    Description   string
    RoleTemplateID uint      // 组统一角色
    Users         []User      `gorm:"many2many:group_users;"`
    ClusterGrants []ClusterGrant `gorm:"polymorphic:Subject;"`
}

// 授权时优先查用户个人授权，无则查用户所在组的授权
```

## 方案三：Namespace 级别隔离（适合多业务线共用集群）

### 场景

生产集群A 有 3 个业务线：
- 电商团队：namespace = `ecommerce-*`
- 支付团队：namespace = `payment-*`
- 物流团队：namespace = `logistics-*`

**需求：** 电商团队的人只能看电商的 Pod，不能看支付的 Pod。

### 解决：ClusterGrant 增加 NamespaceScope

```
用户：王五（电商开发）
角色：业务开发
授权集群：生产集群A
NamespaceScope：ecommerce-prod,ecommerce-staging   ← 只能看这两个 NS
```

**效果：**
- 王五登录 → 进入生产集群A → 只能看到 `ecommerce-*` 的 Pod
- 支付的 Pod 不显示 → 无法查看 → 无法操作

---

## 对比：三种方案适用场景

| 方案 | 适用人数 | 管理复杂度 | 隔离粒度 | 实施建议 |
|------|---------|-----------|---------|---------|
| **逐个授权** | < 50 人 | 中 | 集群级 | Phase 1 先做，跑通基础 |
| **用户组批量** | 50+ 人 | 低 | 集群级 | Phase 2 加入，降低运维成本 |
| **Namespace 隔离** | 任意 | 高 | 命名空间级 | Phase 3 评估，看业务需求 |

## 关键问答

**Q：如果用户同时属于两个组，权限怎么算？**

取最高权限。例如：
- 组A：viewer
- 组B：operator
- 结果：operator（不降级）

**Q：用户个人授权和用户组授权冲突怎么办？**

个人授权优先。例如：
- 用户组：运维部-operator → 全部集群 operator
- 个人授权：李四 → 生产集群A → viewer（特殊限制）
- 结果：李四在生产集群A 是 viewer，其他集群是 operator

**Q：新员工入职，需要几步完成授权？**

- Phase 1（逐个）：创建用户 → 选择角色 → 逐个勾选集群 → 保存（5 分钟）
- Phase 2（用户组）：创建用户 → 加入用户组 → 自动继承授权（30 秒）

**Q：凭证升级后，已登录用户需要重新登录吗？**

不需要。每次 API 请求都会重新 `SelectCredential`，实时生效。
